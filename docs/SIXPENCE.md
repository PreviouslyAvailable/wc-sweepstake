# SIXPENCE

## Design Operating System Specification

### Version 1.3.0

### Movement: Terrace Modernism

### Purpose: Brand, Product and App Development System

---

## 0. Executive Summary

SIXPENCE is a design operating system for documenting football culture as a living civilisation.

It is not a sports brand system.

It is not a campaign identity.

It is not nostalgia.

SIXPENCE treats football culture as an archive of artefacts, rituals, records, formats, voices, systems, diagrams, memories and collected objects. Every output behaves as if it belongs to a fictional governing body's official archive: issued, numbered, dated, annotated, priced, collected and never fully retired.

The system is designed for both humans and machines.

For humans, it provides a conceptual framework: worldview, principles, lineage, refusals, governance and creative logic.

For machines, it provides an ontology: registers, artefact types, signatures, tokens, rules, validation logic and decision structures that can be implemented in an app, content management system, design tool, brand generator or component library.

The system's core belief is:

**Identity is not created through expression. Identity is created through accumulation.**

A single asset does not define SIXPENCE. The archive does.

A card matters because it belongs to a set.
A programme matters because there are decades of programmes.
A stamp matters because it commemorates something.
A diagram matters because it records a system.
A quote matters because it is filed, attributed and preserved.
An app screen matters because it is not just a screen — it is an artefact in the archive.

# 1. Brand Constitution

## 1.1 Core Belief

Culture is built through accumulation, documentation, ritual and shared memory.

SIXPENCE does not perform authority through polish. It gains authority by behaving like something that lasts.

The brand should feel as though it has existed before the current moment and will continue after it. It should not look "new" in the conventional product-design sense. It should feel issued, kept, revised, amended, stamped, torn, filed and rediscovered.

## 1.2 Subject

The subject is not football.

The subject is football culture.

SIXPENCE documents the civilisation around the game:

* programmes
* tickets
* cards
* stickers
* fixtures
* scoreboards
* match reports
* chants
* diagrams
* formations
* instructions
* rituals
* grounds
* tunnels
* seats
* nets
* patents
* local print
* fan graphics
* terrace language
* collected memory

The game is the source.
The culture is the subject.
The archive is the form.

## 1.3 Positioning

SIXPENCE never says:

"We are the best."

SIXPENCE says:

"We are part of a culture."

The brand behaves as a custodian, participant and recorder — not a sponsor, advertiser or outsider.

It does not shout over the culture.
It files the culture.
It numbers the culture.
It preserves the culture.
It lets the culture speak back.

## 1.4 Organising Metaphor

SIXPENCE is a living archive.

Every item produced by the system is an archive entry.

Nothing is "content" in the disposable sense. Everything is an artefact.

Every artefact should answer:

* What is it?
* Who issued it?
* When was it issued?
* What set does it belong to?
* What number is it?
* What format does it use?
* What register does it speak in?
* What does it document?
* Why should someone keep it?

## 1.5 Origin Artefact

The symbolic origin is a 1967 Fulham FC official programme priced at sixpence.

This origin matters because it contains the full code:

* institutional but cheap
* mass-produced but now precious
* local but culturally resonant
* printed with constraint
* designed to serve a function
* collected because it survived

SIXPENCE is named after the value of an artefact, not the value of a logo.

## 1.6 Core Model

The model is:

**Community → Culture → Brand**

Never:

**Brand → Audience**

The brand does not install meaning.
It creates the conditions for meaning to accumulate.

Symbols, marks, cards, badges, artefacts and rituals become powerful because people use them, keep them, alter them, trade them, annotate them and remember them.

## 1.7 Doctrine

The core doctrine is:

**Tight system. Loose hand.**

The system must be strict enough to be recognisable and loose enough to feel human.

SIXPENCE should never collapse into corporate consistency. Its references come from volunteer programme editors, printers, fans, collectors, clubs, local papers and unofficial makers. The app must preserve that spirit without becoming arbitrary.

The answer is not to make everything loose.

The answer is to define which axes are law and which axes are free.

# 3. Refusals

A strong system must know what it refuses.

These are not aesthetic dislikes. They are ideological boundaries.

## 3.1 SIXPENCE Refuses Conventional Sports Marketing

Avoid:

* hero worship
* motivational clichés
* peak-performance language
* "greatness" rhetoric
* speed lines
* aggressive gradients
* athlete-as-superhuman imagery
* trophy obsession
* generic victory language
* "built to win" narratives

Reason:

SIXPENCE is not about spectacle. It is about culture.

## 3.2 SIXPENCE Refuses Decorative Nostalgia

Avoid:

* vintage as filter
* fake distressing without structural reason
* retro typography used as costume
* meaningless badges
* invented dates
* decorative stamps with no issue logic
* "old football" pastiche

Reason:

Historical formats are not mood. They are operating systems.

## 3.3 SIXPENCE Refuses Corporate Consistency

Avoid:

* over-templating
* every asset looking identical
* brand-book perfection
* sterile components
* fixed composition rules across all outputs
* removing the human hand
* treating variation as error

Reason:

The archive should be coherent through metadata, format and rules — not sameness.

## 3.4 SIXPENCE Refuses Empty Lifestyle Branding

Avoid:

* vibes without provenance
* culture as aesthetic surface
* generic community language
* unexplained references
* "premium" minimalism with no cultural reason

Reason:

Every decision must be traceable to source material, system logic or archive behaviour.

## 3.5 SIXPENCE Refuses Generic Tech Neutrality

Avoid:

* frictionless blue-and-white SaaS language
* rounded cards everywhere
* floating shadows
* abstract blobs
* "seamless" as default tone
* product UI that erases brand logic

Reason:

The app is not a neutral container. It is part of the archive.

# 4. Governance Philosophy

## 4.1 Law and Loose

SIXPENCE is governed by fixed and free axes.

The fixed axes create recognition.
The free axes create humanity.

## 4.2 Fixed Axes

These are law.

They should be enforced by design review, app validation and system rules.

Fixed axes:

* register selection
* artefact type
* stock selection
* ink selection
* approved ink combinations
* typography roles
* signature requirement
* colophon requirement where applicable
* truth in annotation
* metadata structure
* numbering logic
* collection logic
* texture requirement
* no gradients
* no decorative fake data
* no arbitrary colour
* no full-colour photography except approved exceptions

## 4.3 Free Axes

These are intentionally loose.

They should vary between artefacts, designers, editions and contexts.

Free axes:

* composition
* crop
* scale
* illustration style within approved category
* degree of misregistration
* hand-lettering personality
* amount of whitespace
* typographic rhythm
* image placement
* annotation density
* paper wear
* edition-specific quirks
* local language
* contributor voice
* collectible rarity treatment

## 4.4 Governance Rule

Do not ask:

"Is this consistent?"

Ask:

"Is this governed?"

A SIXPENCE artefact can look different from another SIXPENCE artefact if its register, format, metadata, inks, signatures and logic are correct.

The system should produce family resemblance, not template sameness.

# 6. Registers

Registers are the four volumes of the system.

A register is selected by use case, not taste.

Each artefact must use exactly one primary register.

## 6.1 WHISPER

### Role

WHISPER is for emotion, memory, restraint and premium storytelling.

Use for:

* brand campaigns
* manifestos
* hero moments
* emotional storytelling
* quiet product storytelling
* premium partnerships
* reflective editorial moments

### Behaviour

WHISPER is quiet, spacious and typographic.

It uses:

* vast negative space
* one image or one message
* editorial serif
* duotone halftone
* minimal composition
* spare copy
* no visual noise

### Required Features

```yaml
whisper:
  required:
    - stock: sixpence_cream
    - max_inks: 2
    - type_display: editorial_serif
    - whitespace_minimum: 60_percent
    - photography_treatment: halftone_duotone
  allowed_signatures:
    - italic_intruder
    - halftone_duotone
    - colophon
  forbidden:
    - perimeter_chant
    - signal_yellow
    - whistle_red
    - full_bleed_colour_fields
    - pixel_grid
```

### Voice

Spare. Declarative. Emotionally loaded.

Example voice:

"Leave with nothing left."

"Some grounds remember everything."

"Filed under courage."

---

### Role

ROAR is for urgency, matchday energy and public broadcast.

Use for:

* fixtures
* results
* announcements
* live moments
* scoreboards
* signage

### Behaviour

ROAR is loud, typographic and immediate.

It uses:

* full-bleed colour

* repetition as texture

### Required Features

# 8. Collection Logic

## 8.1 Principle

The set comes before the item.

A single artefact should imply a larger system.
A collection should create desire through incompleteness.

## 8.2 The Album Is The Product

The album is a first-class artefact.

This means that for any serial content type, designers and product teams must define:

* the collection name
* the total number of items
* the item numbering logic
* the empty slot state
* rarity rules
* completion states
* reissue rules
* archive behaviour
* how users collect, trade, unlock, revisit or complete the set

## 8.3 Collection Schema

```yaml
collection:
  id: string
  name: string
  description: string
  collection_type:
    enum:
      - cards
      - stamps
      - programmes
      - quotes
      - badges
      - fixtures
      - moments
      - plays
      - profiles
  total_items: integer
  issued_items: integer
  numbering_format: string
  rarity_model:
    enum:
      - none
      - common_uncommon_rare
      - numbered_limited
      - foil_special
      - archival_reissue
  empty_slot:
    required: true
    visual_style:
      enum:
        - outlined
        - perforated
        - ghosted
        - stamped_missing
  completion_state:
    enum:
      - incomplete
      - complete
      - archived
      - reissued
```

## 8.4 Rarity Logic

Rarity must be expressed through system behaviour, not arbitrary shine.

Allowed rarity signals:

* issue number
* limited edition count
* foil border in ARCADE only
* alternate stock
* commemorative stamp value
* annotation density
* reissue mark
* archived status
* hand-numbered variant
* misprint status

Forbidden rarity signals:

* arbitrary glow
* luxury gradients
* fake scarcity
* gold without meaning
* premium effects outside collection logic

# 9. Time and Archive Behaviour

## 9.1 Principle

Time is a design material.

SIXPENCE artefacts should gain meaning as they age.

The archive is not static. It is additive and transformative.

## 9.2 Archive States

```yaml
archive_status:
  enum:
    - draft
    - issued
    - amended
    - reissued
    - archived
    - retired_visible
    - lost
    - recovered
```

## 9.3 Ageing Rules

Artefacts may acquire:

* annotations
* amendments
* issue stamps
* reprint marks
* corrections
* provenance notes
* owner marks
* fold lines
* wear
* version history
* completion records

## 9.4 Never Fully Delete

In app development, deletion should be rare.

Where possible, use archival state changes instead:

* hidden from current view
* marked as superseded
* filed as retired
* reissued with amendment
* preserved in collection history

## 9.5 Reissue Logic

```yaml
reissue:
  required_fields:
    - original_issue_id
    - reissue_date
    - reissue_reason
    - visible_reissue_mark
    - version_number
  rules:
    original_remains_accessible: true
    reissue_must_not_overwrite_original: true
```

# 11. Design Tokens

## 11.1 Colour Philosophy

SIXPENCE uses stocks and inks, not palettes.

A stock is what you print on.
An ink is what you print with.

Colour is identity, not decoration.

Rules:

* choose one stock per artefact
* choose one to three inks per artefact
* no gradients
* no arbitrary tints
* no pure white grounds
* no decorative colour
* red is either stock or signal, never both
* colour combinations must be approved

## 11.2 Stocks

```yaml
stocks:
  sixpence_cream:
    hex: "#F1EAD8"
    role: "default paper"
    meaning: "aged programme stock"
    registers:
      - whisper
      - archive

  vermilion_stock:
    hex: "#C8402F"
    role: "special issue paper"
    meaning: "red programme cover; cup tie; legend card"
    registers:
      - archive
    rules:
      print_with:
        - patent_black
        - sixpence_cream
      never_with:
        - whistle_red

# 13. Signature Devices

Every artefact must use at least one signature device.

## 13.1 Italic Intruder

Definition:

Exactly one letter set in italic inside an otherwise roman headline word.

Meaning:

A printing-era accident made deliberate.

    - decorative_random_use
```

Patent-style documentation: figure numbers, leader lines, mono labels and ruled boxes.

Football explained like an invention.

## 13.4 Perimeter Chant

# 20. Kill Tests

Use these during critique, QA and automated validation.

## 20.1 Culture Test

Question:

"Is this football culture or sports marketing?"

Fail if:

* it uses generic performance language
* it centres celebrity over culture
* it feels like an ad instead of an artefact

## 20.2 Archive Test

Question:

"Would someone keep this?"

Fail if:

* it feels disposable
* it has no issue logic
* it has no metadata
* it does not belong to a set, archive or event

## 20.3 Photocopier Test

Question:

"If ten artefacts from the set were photocopied, would they become indistinguishable?"

Fail if:

* all compositions are identical
* the human hand has disappeared
* the system feels corporate

## 20.4 Law and Loose Test

Question:

"Are the right things fixed and the right things free?"

Fail if:

* colours are random
* metadata is fake
* annotations are decorative
* or, conversely, composition is over-controlled

## 20.5 Helvetica Test

Question:

"If the headline face were swapped for Helvetica, would the piece still be recognisably SIXPENCE?"

Fail if:

* only the logo makes it identifiable
* structure is weak
* signatures are absent

## 20.6 Collection Test

Question:

"Does the item imply a larger set?"

Fail if:

* it is not numbered
* no album or collection logic exists
* no empty slot or completion state is considered

## 20.7 Time Test

Question:

"Can this artefact age?"

Fail if:

* it can only exist as a current asset
* it has no archive state
* it cannot be amended, reissued or recovered

# 23. Final Principle

SIXPENCE should not feel like a brand applied to football.

It should feel like football culture finally became legible as a system.

The app should not simply display artefacts.

The app should issue them, collect them, annotate them, age them, refile them, number them and preserve them.

The brand is not the mark.

The brand is the archive.

The system succeeds when a user does not feel they are using a product.

They feel they are holding a piece of the game.
